# Agent Vault — Phase 1: Data Layer

> Status: implementation plan — derived from
> [2026-05-13-agent-vault-and-egress.md](./2026-05-13-agent-vault-and-egress.md)
> (the architecture plan; read first).
> Scope: schema + org-lifecycle code only. No crypto, no vault daemon,
> no container changes, no docker-network changes.

## Goal

Land the Postgres data model that Phase 3 (vault daemon) and Phase 4
(container init) will build on top of. After this PR, the DB has all
the structures vault will eventually fill — but nothing actually
generates DEKs, signs JWTs, or talks to containers yet.

## What this PR delivers

1. Vault-wide `specifyr_vault` schema with `jwt_signing_key` table
   (empty; vault daemon will populate it in Phase 3).
2. Dynamic per-org schema creation: `CREATE SCHEMA org_<id>` + 7
   tables (`service_credentials`, `agent_specs`, `agent_spec_secrets`,
   `agent_sessions`, `master_keys`, `auth_nonces`,
   `secret_access_log`) + per-org Postgres role with grants.
3. `orgs` table gains `bridge_subnet` (cidr) and `init_status`
   (`pending_vault_init` | `ready`) columns.
4. Bridge-subnet allocator: picks next free `/24` from
   `SPECIFYR_BRIDGE_POOL` (default `10.20.0.0/14`).
5. Org-create wired up to allocate subnet + create per-org schema +
   set `init_status='pending_vault_init'`. Vault HTTP call is **not**
   wired yet (Phase 3); the status stays pending for new orgs.
6. Agent-start guard: refuse to spawn if owner-org's
   `init_status != 'ready'`. Backfill migration sets every
   pre-existing org to `'ready'` so today's workflows keep working.

## What this PR does NOT do

- Generate DEKs, JWT signing keys, or any crypto material (Phase 3).
- Implement the KEK provider interface (Phase 3).
- Stand up the vault daemon or `/internal/orgs/<id>/init` (Phase 3).
- Touch `src/runners/company-network.js` or the docker network setup
  (Phase 4+).
- Touch the container-init binary or nftables (Phase 4).

## Task order

1. **Schema declarations** in `schema.ts` (no migration yet — just
   types). Add `cidr` import from `drizzle-orm/pg-core` if available;
   otherwise use `text('bridge_subnet')` with a `::cidr` cast in raw
   SQL (verify before deciding — see "Open questions").
2. **Bridge-subnet allocator** + tests. Pure function, no DB schema
   dependency yet; can be tested with an in-memory list.
3. **Per-org-schema builder** + tests. Use a Postgres test container
   or testing-against-the-dev-DB pattern that the repo already uses
   (mirror whatever existing migration tests do).
4. **`drizzle-kit generate`** — produces `0005_*.sql`. Inspect the
   diff carefully; the vault-wide `specifyr_vault` schema needs
   `CREATE SCHEMA IF NOT EXISTS specifyr_vault` to come **before**
   the `jwt_signing_key` table.
5. **Hand-write `0006_*.sql`**: backfill existing orgs. Pseudo-SQL:
   ```sql
   UPDATE orgs SET init_status = 'ready' WHERE init_status IS NULL;
   -- allocate subnets for orgs that don't have one yet (call the
   -- allocator-equivalent in plain SQL via a CTE, or write a
   -- one-shot Node script — whichever is simpler)
   ```
6. **Extend `createOrgWithAdmin`**: call allocator, insert org row,
   call `createOrgSchema`, all inside the same transaction. **Skip
   the vault HTTP call** — that's Phase 3. Status stays pending.
7. **Agent-start guard**: one early `if (org.init_status !== 'ready')
   throw createError({ statusCode: 503, ... })`. Decide which file —
   probably `start.post.ts`, before any heavy work.
8. **Tests**: end-to-end org-create, guard refuses pending, guard
   allows ready.
9. **Manual smoke test**: `pnpm dev` → create org via UI →
   `psql` and verify `org_<uuid>` schema + 7 tables + role exist;
   `init_status='pending_vault_init'`.

## Tests (concrete)

- `allocator picks the first /24 when DB has no orgs`
- `allocator skips already-used subnets`
- `allocator throws when pool exhausted` (configure tiny pool for the test)
- `allocator serialises concurrent calls — two parallel calls return distinct subnets`
- `createOrgSchema creates exactly the 7 expected tables + role`
- `createOrgSchema grants on secret_access_log include INSERT but not UPDATE/DELETE` (RLS / append-only invariant)
- `createOrgSchema is idempotent` (re-running on existing schema is a no-op or clean error)
- `createOrgWithAdmin leaves init_status='pending_vault_init'`
- `agent-start refuses when init_status != 'ready' with 503`
- `agent-start allows when init_status='ready'` (use a fixture org)
- `migration 0006 sets all pre-existing orgs to 'ready' and allocates their bridge_subnet`

## Open questions to resolve during implementation

- **Drizzle `cidr` / `inet` types**: check what `drizzle-orm/pg-core`
  ships. If neither exists, fall back to `text` columns + raw-SQL
  casts and add a Zod check. Decide once when you first hit it; don't
  speculate ahead.
- **Per-org tables in `schema.ts`**: do we want the type-level
  definitions there even though the tables live in dynamic schemas,
  or only inside `per-org-schema.ts`? Recommendation: declare them in
  `schema.ts` with a `pgSchema('org_template')` wrapper so types are
  reusable; the actual DDL is run against `org_<id>` via a builder.
- **`SPECIFYR_BRIDGE_POOL` default**: confirm `10.20.0.0/14` doesn't
  collide with any subnet the dev compose stack already uses. Check
  `docker-compose.yml`, `docker network ls`. If it clashes, pick a
  different range and update the architecture plan.
- **Existing orgs subnet backfill**: how many orgs exist in dev/prod
  today? If few (< 10), backfill in a one-shot Node script run from
  the migration. If many, write a proper SQL allocator.
- **Test DB strategy**: does the repo use testcontainers, a shared
  dev DB, or per-suite migrations? Mirror existing patterns; don't
  invent a new one for this PR.
