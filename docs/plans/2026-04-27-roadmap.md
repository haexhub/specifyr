## North-star vision (binding)

The user is building paperclip-style AI-companies. Two phases:

1. **Spec phase** — define the company AND its ticket source via spec-kit
2. **Execution phase** — tickets flow into queues; agents work; CEO is
   always informed of completions

UI is project-first: pick a project, then choose Speckit-View (edit
specs) or Runtime-View (live state + history + org chart).

Durable architectural commitments (see also `~/.claude/projects/-home-haex-Projekte-haex-corp/memory/architecture_decisions.md`):

- **Reporting DAG**: workers always report to ≥CEO; `also_notify` for
  peer reports; spec-load enforces DAG (no cycles)
- **Living specs at next-pickup**: spec edits apply on next task
  dispatch, not mid-task
- **Files = truth, SQLite = index**: `<project>/.specops/<slug>/state.db`
  is regenerable from on-disk md/yaml/jsonl files
- **No Kafka**: append-only JSONL events
  (`<project>/.specops/<slug>/events/YYYY-MM-DD.jsonl`); NATS JetStream
  is the pre-approved escalation if real fanout becomes needed

## Near-term — well-scoped, ready to plan in detail

### 10a. Event log + reporting DAG (NEW — vision-foundation)

The reporting model from the vision needs structural support BEFORE
notification transports (which are presentation, not infrastructure).
Concretely:

- **DAG validation in `spec-loader.js`**: build the union graph of
  `reports_to` + new `also_notify` field, reject cycles at load time.
  ~20 lines of code.
- **JSONL event log**: append-only `<project>/.specops/<slug>/events/YYYY-MM-DD.jsonl`.
  Every dispatch-started, dispatch-completed, dispatch-failed,
  approval-{requested,decided}, agent-{spawned,exited} event lands here.
- **Auto-report on completion**: on worker `status === "completed"`, the
  runtime emits a structured report event addressed to CEO + every role
  in the worker's `also_notify` list. CEO receives via its own queue
  (special "report" ticket type, distinguishable via `source` field).
- **`source` field on tickets**: `user` | `agent:<role>` | `ingestor:<name>`.
  Cheap, decided here, used later by UI.

This is the structural layer the UI Runtime-View will read from. Effort:
~3-4h. Sequence-critical.

### 10c. Supervisor / Watchdog ✅ done (2026-04-28)

Implementiert in `src/core/supervisor.js`. In-memory pending tracking via
runtime-event-subscription, default SLA 1h, per-agent override via
`agent.sla_seconds`. Auf SLA-Verletzung: `agent-stuck`-Event im JSONL Log
+ Intervention-Ticket nach `queue-ceo/` mit `source: supervisor` und
strukturiertem `intervention:` Block.

Auto-instanziiert von `CompanyRuntime`; deaktivierbar via `{ supervisor: null }`.

Spätere Erweiterungen (nicht blocking):
- Iteration-Limit-Enforcement aus Task-YAML `termination:`-Block
- `agent.intervene_via: <role>` für hierarchische Sub-Supervisor in größeren Orgs
- Missed-handoff-Detection (`delivers_to`-Edge ohne Folge-Dispatch innerhalb erwarteter Pickup-Zeit)

### 10b. SQLite project index (NEW — derived view layer)

After 10a we have the canonical event log on disk. This inkrement adds:

- `<project>/.specops/<slug>/state.db` — per-project SQLite database
- Tables: `tickets`, `events`, `agent_status`, `approvals`
- **Rebuild-from-disk function** is the deliverable's correctness
  contract — drop the DB, replay all JSONL + scan all queue YAMLs,
  result is identical
- Write-through pattern: every file write is followed by a SQLite
  upsert in the same code path. Crashes between leave at most a missing
  index entry, never lost data.

Effort: ~3-4h. Unblocks UI #13.

### 10. Hermes-side catalog HTTP-MCP wiring (deferred from 7.4)

The MCP server is up; the worker container has `COMPANY_OPS_TOKEN` and
`COMPANY_OPS_URL` in env. **Hermes itself doesn't yet know to use
them.** Today the CEO-prompt would have to manually `curl` the
endpoint via a `code_execution` skill (Inkrement 8's fallback path).

What this inkrement needs:
- Update `catalog/tools/company-ops.yml` from `transport: stdio,
  command: node` to `transport: http, url_env: COMPANY_OPS_URL,
  auth_env: COMPANY_OPS_TOKEN`
- Verify Hermes 0.11 supports HTTP-MCP transport with bearer auth (if
  not — write a thin stdio bridge inside the container that proxies
  to HTTP)
- Confirm via the multi-agent E2E that CEO calls dispatch through the
  proper tool interface, not curl

Effort: depends entirely on Hermes-side support discovery. **0.5h if
HTTP-MCP works out of the box; 4-6h if a sidecar bridge is needed.**

### 11. Notification transports for ApprovalService

`CapabilityApprovalService` already has the policy + Promise/timeout
plumbing. The transport interface is a no-op today. This inkrement
plugs in real channels.

Scope (pick one or two for v1):
- **Signal** via `signal-cli` — practical for solo-dev, sends to your
  own number
- **Telegram** bot — easy API, easy on Linux without GUI
- **Email** via SMTP/nodemailer — universal but slow round-trip
- **Github issue** — async, audit-trail-friendly

Out of scope in v1: Slack, Discord, WhatsApp (Hermes has it but
pairing is fiddly).

Each channel adds a `src/transports/<channel>.js` that satisfies
`{ async notify({channel, payload}) }`. Plus a config schema for
secrets/setup. ~2-3h per channel including tests.

### 12. Memory limits per agent (OOM defense)

The E2E was flaky once due to system memory pressure (exit 137,
detailed in commit `2ea60be`). Hermes containers can OOM under
load. Mitigations:

- Set `--memory=1g` (or per-agent override via `agent.resources.memory`)
  in the runner's baseline flags so OOM is bounded and predictable
- Capture `docker events` for OOM-kill so we surface it as a typed
  error, not silent SIGKILL

Smallest scope: ~1h. Already plumbed via existing `resources` field.

### 13. UI: project-first navigation + Speckit-View / Runtime-View

The vision-binding UI shape: landing page = project picker; inside a
project, two parallel views.

**Speckit-View** (refines existing UI): edit constitution, agent specs,
ticket-source spec. Already partially present.

**Runtime-View** (new): three logically-distinct panes that must NOT
mix concerns:
- **Org chart** — static from spec (`reports_to` tree visualised)
- **Live status** — pulled from `/api/projects/<slug>/company/status`
  (who's currently dispatched, queueDepth per role)
- **History** — read from SQLite (#10b); time-ordered event log with
  filters per agent / per ticket

Drilling into one agent shows current task + that agent's spec docs
+ approval inbox for that agent.

Backend is mostly there (start/stop/status/authorize/agents/dispatch).
Effort: ~6-10h depending on Vue component depth. Builds on #10a + #10b.

### 14. `docker compose up` E2E

Today's E2E test runs against a local CompanyRuntime instance with
local docker. That doesn't exercise the haex-corp-in-container path.
We need a separate integration that:

- Starts haex-corp via `docker compose up` in detached mode
- POSTs `/api/projects/<slug>/company/start` from the host
- Verifies the spawned hermes container ALSO uses correct host paths
  (validates `HAEX_CORP_HOST_PROJECT_ROOT` plumbing from inkrement 6.2)
- Tears down compose

Catches the docker-out-of-docker class of bugs that the unit-level E2E
can't see. ~2h.

### 14b. Ingestor framework (NEW — own track)

Vision: each project specifies its own ticket source(s) — GitHub/GitLab
issues, Slack/Signal/email chat with CEO, another speckit workflow,
manual UI form. Each source has an approval mode (auto-execute /
human-confirm-first).

**MVP scope: manual UI form only.** Ship this with #13 so the UI is
useful end-to-end before adding network-facing ingestors.

Post-MVP, each source is a `src/ingestors/<name>.js` module:
- Uniform interface: emit ticket-shaped events with `source`/`approval_mode`
- Auth via existing secrets resolver pattern (GitHub PAT, Slack token, …)
- Per-source config in the project's spec dir, edited via spec-kit
  alongside agent specs
- Approval-mode honored: `auto` → goes straight into queue; `confirm`
  → goes into a pending-tickets pool, UI shows it for OK

Pick the first ingestor based on user demand: GitHub-issues is the
most leverage for an open-source dev workflow. Effort per ingestor:
~3-5h including auth, polling/webhook, filter rules, tests.

## Mid-term — needs design first

### 15. Spec-kit-extension naming polish

Currently spec-kit emits compatibility warnings on install:
```
Command 'speckit-company.init' does not follow pattern 'speckit.{ext}.{cmd}'.
Registering as 'speckit.speckit-company.init'.
```

Fix options:
- **A)** Rename slug from `speckit-company` to `company`. Commands
  become `speckit.company.X`, no warnings, **but** the GitHub URL,
  folder name, registration key all need updating
- **B)** Rename commands to `speckit.speckit-company.X` (canonical).
  Ugly redundancy but matches spec-kit's pattern exactly
- **C)** Stay where we are. Warning is cosmetic; aliasing works

Trade-off discussion + decision needed before code change. ~1-2h
including all references.

### 16. Backup / restore tooling

Naturally follows Inkrement 9 (data separation). Once data is in its
own tree, scripts to:
- `pnpm backup:data` → tar + timestamp under `~/.local/state/haex-corp-backups/`
- `pnpm restore:data <archive>` → safety-checked restore

Estimate: 2h. Wait until 9 lands so we backup the right thing.

### 17. CI workflow expansion

Today CI builds the hermes-agent image and pushes to GHCR. Missing:
- Run unit tests + smoke tests on PR
- Run E2E with a mocked LLM (or skip when no API key)
- TypeScript typecheck (currently no script — `pnpm tsc --noEmit`?)

~2h.

### 18. Approval-audit retention

EventStore today appends approvals; no rotation, no GC. For long-running
companies this grows unbounded. Define a retention policy + sweep
script.

Could be a 1h fix or a multi-day project depending on whether we want
queryable history (SQLite migration?) or just rolling files.
