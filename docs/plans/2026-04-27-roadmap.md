## Near-term — well-scoped, ready to plan in detail

### 10. Hermes-side catalog HTTP-MCP wiring (deferred from 7.4)

The MCP server is up; the worker container has `COMPANY_OPS_TOKEN` and
`COMPANY_OPS_URL` in env. **Hermes itself doesn't yet know to use
them.** Today the CEO-prompt would have to manually `curl` the
endpoint via a `code_execution` skill (Inkrement 8's fallback path).

What this inkrement needs:
- Update `catalog/tools/company-ops.yml` from `transport: stdio,
  command: node` to `transport: http, url_env: COMPANY_OPS_URL,
  auth_env: COMPANY_OPS_TOKEN`
- Verify Hermes 0.11 supports HTTP-MCP transport with bearer auth (if
  not — write a thin stdio bridge inside the container that proxies
  to HTTP)
- Confirm via the multi-agent E2E that CEO calls dispatch through the
  proper tool interface, not curl

Effort: depends entirely on Hermes-side support discovery. **0.5h if
HTTP-MCP works out of the box; 4-6h if a sidecar bridge is needed.**

### 11. Notification transports for ApprovalService

`CapabilityApprovalService` already has the policy + Promise/timeout
plumbing. The transport interface is a no-op today. This inkrement
plugs in real channels.

Scope (pick one or two for v1):
- **Signal** via `signal-cli` — practical for solo-dev, sends to your
  own number
- **Telegram** bot — easy API, easy on Linux without GUI
- **Email** via SMTP/nodemailer — universal but slow round-trip
- **Github issue** — async, audit-trail-friendly

Out of scope in v1: Slack, Discord, WhatsApp (Hermes has it but
pairing is fiddly).

Each channel adds a `src/transports/<channel>.js` that satisfies
`{ async notify({channel, payload}) }`. Plus a config schema for
secrets/setup. ~2-3h per channel including tests.

### 12. Memory limits per agent (OOM defense)

The E2E was flaky once due to system memory pressure (exit 137,
detailed in commit `2ea60be`). Hermes containers can OOM under
load. Mitigations:

- Set `--memory=1g` (or per-agent override via `agent.resources.memory`)
  in the runner's baseline flags so OOM is bounded and predictable
- Capture `docker events` for OOM-kill so we surface it as a typed
  error, not silent SIGKILL

Smallest scope: ~1h. Already plumbed via existing `resources` field.

### 13. UI: company-runtime control panel

Backend is fully usable from curl now. Make it clickable:

- Start/stop buttons per project (already have endpoints)
- Live status: running/idle, queueDepth, agent roster (pull
  `/company/status`)
- Approval inbox: pending requests with approve/deny buttons
- Recent dispatched-events stream (SSE bridge similar to run-start)

Effort: ~4-6h depending on Vue component depth. Pure frontend work.

### 14. `docker compose up` E2E

Today's E2E test runs against a local CompanyRuntime instance with
local docker. That doesn't exercise the haex-corp-in-container path.
We need a separate integration that:

- Starts haex-corp via `docker compose up` in detached mode
- POSTs `/api/projects/<slug>/company/start` from the host
- Verifies the spawned hermes container ALSO uses correct host paths
  (validates `HAEX_CORP_HOST_PROJECT_ROOT` plumbing from inkrement 6.2)
- Tears down compose

Catches the docker-out-of-docker class of bugs that the unit-level E2E
can't see. ~2h.

## Mid-term — needs design first

### 15. Spec-kit-extension naming polish

Currently spec-kit emits compatibility warnings on install:
```
Command 'speckit-company.init' does not follow pattern 'speckit.{ext}.{cmd}'.
Registering as 'speckit.speckit-company.init'.
```

Fix options:
- **A)** Rename slug from `speckit-company` to `company`. Commands
  become `speckit.company.X`, no warnings, **but** the GitHub URL,
  folder name, registration key all need updating
- **B)** Rename commands to `speckit.speckit-company.X` (canonical).
  Ugly redundancy but matches spec-kit's pattern exactly
- **C)** Stay where we are. Warning is cosmetic; aliasing works

Trade-off discussion + decision needed before code change. ~1-2h
including all references.

### 16. Backup / restore tooling

Naturally follows Inkrement 9 (data separation). Once data is in its
own tree, scripts to:
- `pnpm backup:data` → tar + timestamp under `~/.local/state/haex-corp-backups/`
- `pnpm restore:data <archive>` → safety-checked restore

Estimate: 2h. Wait until 9 lands so we backup the right thing.

### 17. CI workflow expansion

Today CI builds the hermes-agent image and pushes to GHCR. Missing:
- Run unit tests + smoke tests on PR
- Run E2E with a mocked LLM (or skip when no API key)
- TypeScript typecheck (currently no script — `pnpm tsc --noEmit`?)

~2h.

### 18. Approval-audit retention

EventStore today appends approvals; no rotation, no GC. For long-running
companies this grows unbounded. Define a retention policy + sweep
script.

Could be a 1h fix or a multi-day project depending on whether we want
queryable history (SQLite migration?) or just rolling files.

## Suggested sequencing

A reasonable order, optimising for "user-visible win per session":

1. **Inkrement 8** (multi-agent dispatch) — closes the orchestration
   loop. Without it, the architecture is half-built.
2. **Inkrement 12** (memory limits) — small, defends against the
   E2E-flake we already saw.
3. **Inkrement 13** (UI) — turns "yes the backend works" into "I can
   click it". Big motivation boost.
4. **Inkrement 9** (data separation) — clean refactor; better to do
   AFTER the UI exposes data dirs visually so we can tell what
   migrated.
5. **Inkrement 11** (notification transport, pick 1) — Signal or
   Telegram first; makes approvals actually usable beyond the UI.
6. **Inkrement 10** (Hermes HTTP-MCP wiring) — depends on 8's curl
   fallback already proving the call path.
7. **Inkrement 14** (compose E2E) — confidence in deploy story.
8. **Inkrement 15** (slug rename / naming polish) — cosmetic; do
   when the rest is stable.
9. Rest in mid-term order.

Alternative: if you want to PUBLISH first, prioritise 14 + 17 (CI) +
slug-polish, then 8 makes a good "v0.2 launch" story.
